# 通用极简编码规范

## 简单优先

存在多个实现方案时，优先选择最简单、最直接、最容易维护的方案。

- 只满足当前明确需求，不为未来可能的变化提前设计。
- 只写解决问题所需的代码，不主动增加未被要求的灵活性或配置项。
- 只有明确复用、独立职责或可读性收益时才抽象。
- 避免仅用于转发、包装或统一形式的中间层。

不要为了“更优雅”而主动引入 `Service`、`Repository`、`Adapter`、`Mapper`、`ViewModel`、`DTO`、`Factory`、`Builder`、`Strategy` 或 `Wrapper`。项目已有架构或当前需求确实需要时，继续遵循项目约定。

## 逻辑就近

一段逻辑只使用一次且足够短时，优先放在使用位置附近。约 20 行只是判断是否值得抽取的参考，不是硬性限制。

- 不要为了提取而提取，也不要为了封装而封装。
- 相关代码尽量放在一起，减少不必要的跨文件跳转。
- 不主动创建只承载一两个一次性函数或常量的辅助文件。
- 共享逻辑真实存在后，再创建 `helpers`、`utils`、`constants`、`mapper` 或 `adapter` 等模块。

## 命名与注释

- 命名应表达具体业务含义，避免 `processData`、`handleData`、`formatData`、`transformData` 等泛化词。
- 注释只说明业务规则、设计原因、特殊限制或不直观的复杂分支，不复述代码本身。

## 最小修改

修改已有代码时，优先保持原有结构并控制 diff 范围。

- 不因一个小需求重构整个页面、模块或目录。
- 不顺手优化无关代码、注释、命名或格式。
- 只清理由本次改动产生的未使用项，不擅自删除既有死代码。
- 每一行改动都应能直接追溯到当前需求。

## 保持项目一致性

优先遵循当前项目已有的目录、命名、代码组织和实现风格。一致性优先于个人偏好；不要仅以“最佳实践”为由改写正常工作的既有结构。

## 实现前自检

- 能否减少文件、函数、层级或包装？
- 是否为尚未出现的需求提前设计？
- 是否存在只调用一次的辅助函数或只有转发作用的抽象？
- 是否修改了与当前任务无关的代码？
- 删除新增抽象后，代码是否更容易理解？

优先选择简单代码而不是漂亮架构，选择容易理解而不是高度抽象，解决今天真正需要的问题而不是未来可能出现的问题。
