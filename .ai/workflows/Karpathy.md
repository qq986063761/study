# Karpathy 行为指南

用于减少 LLM 常见编码错误。可与项目专属指令合并使用；遇到更具体的项目规则时，同时遵守更具体的约束。

**取舍：** 这些指南更偏向谨慎而非速度。简单任务应结合实际判断，不增加无必要的流程。

## 1. 编码前先思考

**不要想当然，不要掩盖困惑，要显式说明关键假设和权衡。**

在实现前：

- 明确影响实现的假设；无法从代码和上下文确认且不同选择会显著改变结果时，再向用户提问。
- 如果存在多种合理解释，说明差异和选择依据，不默默引入额外范围。
- 如果有明显更简单的方法，直接指出并优先采用。
- 先确认目标、边界和成功标准，再开始修改。

## 2. 简单优先

**只写解决问题所需的最少代码，不做猜测性扩展。**

- 不实现超出需求范围的功能。
- 不为一次性代码做抽象。
- 不添加未被要求的灵活性或可配置性。
- 不为缺乏现实依据的场景堆叠错误处理。
- 如果实现明显超过问题本身所需的复杂度，继续精简。

问自己：“资深工程师会觉得这段实现过度复杂吗？”如果会，就继续简化。

## 3. 最小范围改动

**只改必须改的部分，只清理由本次改动造成的问题。**

编辑现有代码时：

- 不顺手优化相邻代码、注释或格式。
- 不重构没有问题的内容。
- 保持现有风格，即使个人偏好不同。
- 发现无关问题时可以指出，但不要擅自修改。
- 删除由本次改动导致未使用的 import、变量或函数，不扩大清理范围。

检验标准：每一行改动都应能直接追溯到当前任务。

## 4. 目标驱动执行

**先定义可验证的成功标准，再迭代到验证通过。**

把任务转化为可验证目标，例如：

- “添加校验” → 覆盖非法输入并确认预期行为。
- “修复 bug” → 先稳定复现，再确认修复后不再出现。
- “重构 X” → 确认重构前后外部行为一致。

多步骤任务使用简短计划，并为每一步写清验证项。验证方式服从项目自身的执行约束。

这些指南有效的标志是：diff 中不必要改动更少，因过度复杂导致的返工更少，关键歧义在实现前得到处理。
