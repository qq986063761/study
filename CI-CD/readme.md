# CI/CD 介绍

## 1. 什么是 CI / CD

- **CI（Continuous Integration，持续集成）**  
  指开发者把代码频繁提交到共享仓库后，自动运行构建、测试、代码检查等流程，尽快发现问题。

- **CD（Continuous Delivery / Continuous Deployment，持续交付/持续部署）**  
  指将通过测试的代码自动推送到预发布环境或生产环境，实现更快、更稳定的发布。

> 简单理解：
> - CI 负责「确保代码质量」
> - CD 负责「确保代码能稳定上线」

---

## 2. CI 的核心目标

CI 的重点是：

- 代码提交后自动触发流程
- 自动安装依赖
- 自动执行测试
- 自动检查语法、格式、类型错误
- 提前发现集成问题

### 典型流程

1. 开发者提交代码
2. 触发 CI 流程
3. 拉取代码
4. 安装依赖
5. 执行构建
6. 运行单元测试 / 集成测试
7. 生成报告
8. 失败时通知开发者

---

## 3. CD 的核心目标

CD 的重点是：

- 将通过测试的版本自动发布
- 减少人工手动部署步骤
- 提高上线效率和一致性

### 两种常见理解

- **Continuous Delivery（持续交付）**  
  代码可以随时部署，但仍需要人工点击发布。

- **Continuous Deployment（持续部署）**  
  代码一通过测试就自动部署到生产环境。

---

## 4. 一个完整的 CI/CD 流程

```text
开发提交代码
   ↓
触发 CI
   ↓
运行构建 / 测试 / 检查
   ↓
如果通过
   ↓
触发 CD
   ↓
自动部署到测试环境/生产环境
```

---

## 5. 为什么要用 CI/CD

- 提高开发效率
- 减少手动操作错误
- 让每次提交都可验证
- 更快发现问题
- 更容易实现快速迭代
- 提升上线稳定性

---

## 6. 常见工具

- **GitHub Actions**：GitHub 平台内置 CI/CD
- **GitLab CI**：GitLab 自带流水线
- **Jenkins**：老牌 CI/CD 工具，灵活度高
- **CircleCI**：云端 CI/CD 平台
- **Docker + Kubernetes**：常用于部署与发布流程

---

## 7. 一个实际例子

假设你开发一个前端项目：

- 提交代码到 Git 仓库
- GitHub Actions 自动运行：
  - npm install
  - npm run build
  - npm test
- 测试通过后自动部署到服务器或静态托管平台

---

## 8. 简单记忆方式

- **CI = 代码提交后自动检查**
- **CD = 代码通过后自动发布**

---

## 9. 面试时可以这样回答

> CI/CD 是一种自动化流程，CI 负责在代码提交后自动构建和测试，CD 负责在代码通过验证后自动部署到环境中。它的目的是减少人工操作、提高开发效率、降低上线风险。
