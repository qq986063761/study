# command、rules、skills

面向 **AI 辅助编程**（如 Cursor）时，这三类东西解决的是同一件事的不同侧面：**怎么让模型在「对的时候」按「对的规矩」做「对的动作」**。下面分概念说明，并交代它们如何叠在一起用。

## Command（命令）

**是什么**：用户或工具链**主动触发**的一条「去做某件事」的入口——例如在对话里输入**斜杠命令**、在 CLI 里执行子命令、或通过快捷键/面板选用预制动作。本质是**事件驱动的单次调用**：触发 → 执行约定流程（可能附带固定 Prompt、脚本或参数）。

**特点**：
- **显式**：一般由人点选或输入，意图清晰，适合「我现在就要跑这个流程」。
- **短路径**：常把多步操作封装成一步，减少口述细节。
- **与对话的关系**：可以看作在自然语言之外，给模型/运行时一个**稳定的结构化入口**。

## Rules（规则）

**是什么**：写在仓库或用户配置里的**长期、稳定约束**——例如项目的 `.cursor/rules`、全局规范文件等。内容通常是：代码风格、安全红线、目录约定、禁止事项、与业务相关的固定说明。模型在**相关会话或匹配到的文件上下文**中会被注入或参考这些条文。

**特点**：
- **持续性**：不依赖某一句用户话是否提到；像「项目的宪法」或「常驻系统提示」。
- **可分层**：可区分用户级、项目级、路径/语言级（按 glob 生效），便于大仓库分区治理。
- **偏「边界与默认值」**：告诉模型**不要怎样、优先怎样**，减少每次从零复述规范。

## Skills（技能）

**是什么**：以独立文档（常见为 `SKILL.md`）等形式存在的**可发现、按需加载的操作说明**：某一类任务（例如「按本项目规范写 Ajax」「怎么加 i18n」）的步骤、约定、检查清单与示例。运行时往往在**判断任务与某 skill 相关**时再读取，避免把所有细节塞进每条消息。

**特点**：
- **模块化**：按场景拆分，新增或迭代不影响全局 rules 的体积。
- **可复用**：同一 skill 可在多个项目拷贝或共享；社区也有 skill 集市类资源。
- **偏「怎么做」**：比 rules 更**程序化和任务导向**（先读 skill 再动手），比单次 command 更**可展开成多步规范**。

## 三者关系（怎么配）

| 维度 | Command | Rules | Skills |
|------|---------|-------|--------|
| 触发 | 多为用户显式触发 | 上下文匹配或项目默认注入 | 任务相关时由系统/模型选用 |
| 内容侧重 | 单次动作、入口 | 边界、风格、红线 | 某类任务的完整做法 |
| 典型心智 | 「执行这个」 | 「始终遵守」 | 「这类事先看这篇」 |

**实践顺序**：用 **Rules** 管住全局与红线；用 **Skills** 沉淀可重复的专项流程；用 **Command** 把最高频或最标准的路径一键触发。三者互补，而不是互相替代。

# Harness

**定义**：在 AI 研究与工程语境下，**Harness** 指把「被评对象」（模型、Agent、管线）接入**固定环境**的一整套**运行与度量架**：统一数据/提示、调用方式、解码与后处理、评分规则与汇总，使结果**可重复、可对比**。词源来自「马具」——把强而难控的能力**拴进同一套轨道**里运转。

**在理论中的位置**：
- **不是模型本身**，也不等价于 Agent「怎么思考、怎么选工具」的**策略**；Harness 负责的是**实验协议**：给定输入分布与评价函数，输出可审计的指标与轨迹。
- **与基准（Benchmark）的关系**：基准定义「考什么」；Harness 定义「怎么考」——同一基准换不同 Harness（提示模板、是否 CoT、判分脚本）可能得到不可比的结果，因此论文与开源评测常强调 **harness 版本** 与配置。

**常见类型**：
- **评测 Harness（Evaluation Harness）**：对 LLM/VLM 跑选择题、生成题、代码题等；封装数据集加载、模型 API 或本地推理、解析模型输出、与标准答案或裁判模型对齐打分。理论意义是**把「能力」操作化为可复现的随机变量估计**（在固定协议下的表现）。
- **训练/对齐 Harness**：在 RLHF、DPO 等流程里，有时把「奖励模型 + 采样 + 日志」也称 harness，强调**反馈闭环的标准接口**；核心是**策略与信号如何接入**，而非单条 loss 公式。
- **系统/测试 Harness**：对含模型与工具的整条服务做集成测试或回归；在 AI 系统论里对应**可观测、可注入、可断言的运行台架**。

**要点**：
- **可复现性**：随机种子、温度、截断、工具版本固定，否则「同一模型」的分数不可比。
- **泄漏与污染**：训练集与测试集若经同一管线预处理，Harness 需能区分**合法评测集**与**记忆/泄漏**；这是评测理论里**有效性与可信度**的一部分。
- **与 Agent 框架的分工**：LangGraph 等偏**控制流与状态**；Harness 偏**按协议跑完并记账分**——二者可组合（例如用框架跑任务，用 harness 汇总成功率与成本）。

**小结**：在 AI 理论话语里，Harness 把「智能行为」**锚定**到**协议化的观测与指标**上；没有它，讨论「某模型更强」容易沦为不可证伪的定性说法。读论文或跑开源榜时，应同时看**模型、数据、Harness 配置**三件套。

# 核心概念（理论概要）

## LLM（大语言模型）

**定义**：在大规模文本语料上通过自监督（如「预测下一个词」）预训练的 Transformer 类模型，参数量通常达到数十亿级以上，具备语言理解、生成、推理与少量示例下的任务泛化能力。

**要点**：
- **预训练**：从海量数据中学习统计规律与知识片段，不针对单一任务标注。
- **上下文窗口**：单次可处理的 token 长度有限，长文档需截断、摘要或分块（与 RAG 常配合）。
- **幻觉**：模型会生成流畅但不符合事实的内容；缓解手段包括 RAG、工具调用、后处理与人在回路。
- **与下游应用的关系**：可直接「零样本 / 少样本」使用，也可通过 Prompt、微调、RAG 等适配业务。

## Prompt 工程（提示工程）

**定义**：通过设计输入文本的结构、角色、约束、示例与输出格式，引导 LLM 稳定产出期望结果，而无需或尽量少改模型权重。

**常见技法**：
- **角色与任务说明**：明确身份、目标、受众与成功标准。
- **Few-shot**：在提示中给 1～若干正确输入输出示例，诱导模仿格式与推理方式。
- **Chain-of-Thought（CoT）**：要求「逐步推理」，提升算术、逻辑类任务表现（也会更长、更耗 token）。
- **结构化输出**：约定 JSON、字段名、枚举值，便于程序解析（可与 JSON mode / 约束解码配合）。

**与微调的关系**：Prompt 偏「推理时配置」；微调偏「改变参数」；二者常组合：小改行为用 Prompt，深度对齐风格/领域用语可考虑微调或 RAG。

## 微调流程与 LoRA / QLoRA

**全量微调（Full Fine-Tuning）**：更新全部权重，效果好但对算力、显存、数据量要求高，且易灾难性遗忘。

**LoRA（Low-Rank Adaptation）**：
- **思想**：冻结原模型大部分权重，仅在部分线性层旁路插入低秩矩阵 A·B（秩 r 远小于原维度），只训练这部分小参数。
- **效果**：显存与可训练参数大幅下降，便于单卡或多卡做领域适配；推理时可合并权重进原层，几乎不增加延迟。
- **典型流程**：准备指令/对话数据 → 选目标模块（常 attention 的 q/v 等）→ 设 rank、学习率、epoch → 训练 → 合并或保留适配器按需加载。

**QLoRA（Quantized LoRA）**：
- **思想**：基座模型以 4-bit（等）量化加载，降低显存占用；仍用 LoRA 训练适配器，在数值上通过反量化与梯度策略保持可训练性。
- **适用**：显存紧张、希望在消费级 GPU 上微调较大模型。
- **注意**：需关注量化与训练库的兼容性、学习率与秩的选择；极端情况下精度相对全精度+LoRA 可能有差距。

**小结**：微调解决「领域话术、格式、行为」等；通用知识与事实更新更常配合 RAG 或持续预训练，而非单靠小数据微调。

## RAG（检索增强生成，Retrieval-Augmented Generation）

**定义**：在生成前从外部知识库（向量库、文档库、图数据库等）检索与当前问题相关的片段，把检索结果作为上下文一并输入 LLM，使回答**可溯源、可更新**，减轻纯参数记忆带来的幻觉与过时。

**典型流水线**：
1. **索引**：文档切分（chunk）→ 嵌入模型向量化 → 存入向量索引（可辅以元数据过滤）。
2. **检索**：用户问题向量化 → 相似度检索（Top-K）→ 可选重排序（rerank）。
3. **生成**：将「检索到的片段 + 问题 + 系统指令」拼进 Prompt，由 LLM 生成答案；可要求引用片段编号或出处。

**与微调的边界**：RAG 负责「动态、可审计的事实与文档」；微调负责「语气、任务形态、工具使用习惯」等。生产系统常 RAG + 好 Prompt + 可选轻量 LoRA 一起用。

## Agent 基础范式（ReAct、Plan-and-Execute）

**定义**：在 LLM 之上用**循环控制流**把「推理」与「对外部世界的动作」（调用 API、查库、跑代码、点 UI 等）串起来，使模型不只在上下文里「说完即止」，而是**多步达成目标**。RAG 常作为 Agent 的一种**工具**（检索），Agent 则强调**决策与工具编排**。

### ReAct（推理与行动）

**是什么**：*ReAct* 这个名字来自英文 **Reason**（推理）和 **Act**（行动）。这类 **AI Agent** 指：模型在完成任务时**交替**做两件事——先用自然语言**推理**当前局面、下一步该做什么，再发出**可执行动作**（如调用搜索、计算器、数据库等工具）；运行时把工具的返回当作**观察**写回对话，模型再基于新信息继续推理与行动，直到给出最终答案。也就是中文常说的**「想一步、做一步、看结果、再想」**，而不是单轮里一次性「说完」。

**思路**：让模型交替输出**推理片段（Thought）**与**可执行动作（Action）**，环境或运行时返回**观察（Observation）**，再进入下一轮，直到给出**最终答案（Final Answer）**。

**典型循环**：Thought → Action（如调用某工具及参数）→ Observation（工具返回）→ Thought → …

**优点**：单轮 Prompt 难以覆盖的长链条任务可逐步纠偏；动作与观察把**事实**从模型参数里「拉」到当前上下文，减轻幻觉。

**代价与注意**：步数多则延迟与 token 成本高；需约束动作空间（允许的工具、参数模式）、超时与重试，并对危险操作做权限与人审。

### Plan-and-Execute（先规划再执行）

**思路**：分两阶段（或多阶段）：**规划器**先产出结构化计划（子任务列表、依赖、验收标准）；**执行器**按计划逐步调用工具或子 Agent 完成子任务；必要时根据失败或新信息**重规划（replan）**。

**与 ReAct 的对比**：ReAct 更「边想边做」，灵活但易在长任务上漂移；Plan-and-Execute 把「要做什么」先写清，利于**可审计、可并行子任务、人类改计划**；缺点是规划可能脱离实际，需执行反馈闭环。

**实践要点**：计划宜短步、可验证；执行层仍可内嵌小型 ReAct 处理单步不确定性；与 RAG、MCP、函数调用等组合时，明确**谁负责选工具、谁负责解析 Observation**。

**小结**：ReAct 适合工具多、环境反馈快的交互式任务；Plan-and-Execute 适合步骤清晰或可分解的项目型任务。生产上常见**混合**：粗粒度计划 + 细粒度 ReAct 执行 + RAG/工具兜底。

## AI 应用框架与库（LangChain、LlamaIndex、Semantic Kernel）

**共性**：都在 LLM 之上提供**可复用抽象**（链/图、工具、记忆、检索、输出解析），降低「从 Prompt 到可上线系统」的胶水代码量；选型常看语言生态、与向量库/云厂商的集成深度、团队熟悉度。

**LangChain**：
- **定位**：偏**通用编排**：`Runnable`/`LCEL`、Agent、工具绑定、多消息对话、与多种向量库和模型 API 的连接器。
- **要点**：生态大、组件多；版本迭代快，需注意 API 稳定性与文档版本对齐。适合需要灵活拼装链路与 Agent、且希望快速对接多种后端的团队。

**LlamaIndex**：
- **定位**：从名字可见重心在**数据索引与 RAG**：文档加载、切分、索引（向量/关键词/图等）、查询引擎、路由与后处理，也支持 Agent 与工作流。
- **要点**：在「把企业文档变成可问答知识」上抽象较完整；若主战场是检索质量、多数据源融合，常优先考虑；与 LangChain 也可组合使用（按层分工）。

**Semantic Kernel（SK）**：
- **定位**：微软系、**强类型**倾向：Kernel、Plugins（函数即技能）、Planner、Memory；与 .NET / Azure / Copilot 栈结合紧。
- **要点**：适合已在 C# / 企业微软云环境、希望插件化与编排可测试的团队；概念上与其他框架的「工具 + 规划器」可对照理解。

**小结**：**编排与 Agent 通用能力**看 LangChain；**RAG / 索引 / 问答数据管线**看 LlamaIndex；**微软技术栈与插件化**看 Semantic Kernel。三者都不是「模型本身」，而是**工程层**选型。

## 多智能体框架（LangGraph、AutoGen）

**多智能体（Multi-Agent）**：多个可独立决策或分工的 Agent（或带角色的 LLM 实例）通过消息、共享状态或图结构协作，以分解复杂任务、并行子任务或引入「批评—修订」等模式；核心难点是**状态一致性、终止条件、权限与成本**。

**LangGraph**：
- **定位**：常与 LangChain 同生态，用**有向图**表达循环、分支、人机中断（human-in-the-loop）、检查点（checkpoint），适合把 ReAct、Plan-and-Execute、多角色协作**显式画成状态机**。
- **要点**：可控性强、便于调试长流程；需要团队接受「图 + 状态」建模方式。

**AutoGen**：
- **定位**：微软开源的对话式多 Agent 框架，强调**可编程对话模式**：多角色代理、群聊、工具调用、与自定义 Agent 组合；适合快速试验「研究员 + 编码员 + 评审」这类协作剧本。
- **要点**：抽象偏对话与角色；与具体业务持久化、观测（tracing）等仍需自行或结合其他工具补齐。

**小结**：LangGraph 偏**结构化流程与可恢复执行**；AutoGen 偏**对话式角色协作**；二者都可实现多智能体，但建模心智模型不同。

## MCP 与 Dify

**MCP（Model Context Protocol）**：
- **定义**：面向「模型 ↔ 工具/数据源」的**开放协议**（常以 stdio 或 HTTP 等承载），用统一方式描述**工具有哪些、参数模式、如何调用**，让客户端（如 IDE、Agent 运行时）能连接多个 MCP Server，而不用为每个 API 写一套专用适配。
- **要点**：解决的是**工具集成标准化**；不负责业务工作流编排——编排仍由上层应用或 Agent 框架完成。与函数调用（function calling）的关系：MCP 多在**进程/服务边界**暴露能力，函数调用多在**单次补全请求内**绑定 schema。

**Dify**：
- **定位**：开源的 **LLM 应用开发平台**（工作流、知识库/RAG、Agent、可观测与运营），偏**低代码/运营**：可视化编排 Prompt、检索、分支与发布。
- **要点**：适合快速搭对话应用与内部知识助手；深度定制或极复杂控制流时，可能仍需下沉到代码框架（LangChain 等）或自研。

**小结**：MCP 是**协议与集成层**；Dify 是**产品与编排平台层**——层次不同，可并存（例如在 Dify 或自研 Agent 中消费 MCP 暴露的工具）。

## 多模态模型（Multimodal）

**定义**：在预训练或后训练中同时处理**多种模态**（常见为文本 + 图像，也可含音频、视频、文档版面等），学习跨模态对齐，使单次推理能接受多模态输入并生成文本或其他模态输出。

**能力谱系**：
- **理解**：图像描述、视觉问答、图表/界面理解、文档 OCR + 语义理解等。
- **生成**：文生图、图生图等常由扩散模型等承担；「单一多模态大模型」有的侧重**理解 + 指代**，生成能力因模型而异。

**与应用架构的关系**：多模态输入需在管线中做**编码**（图像 URL、base64、专用 vision encoder），并占用更多 token/延迟；RAG 可扩展到**图文混合索引**（多向量、多路检索）。Agent 场景下，多模态常用于**屏幕/截图/设计稿**作为 Observation 的一部分。

**注意**：模态越多，评估与安全面越复杂（隐私图像、版权、提示注入跨模态）；生产上需配额、审计与内容策略。
