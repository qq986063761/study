# 核心概念（理论概要）

## LLM（大语言模型）

**定义**：在大规模文本语料上通过自监督（如「预测下一个词」）预训练的 Transformer 类模型，参数量通常达到数十亿级以上，具备语言理解、生成、推理与少量示例下的任务泛化能力。

**要点**：
- **预训练**：从海量数据中学习统计规律与知识片段，不针对单一任务标注。
- **上下文窗口**：单次可处理的 token 长度有限，长文档需截断、摘要或分块（与 RAG 常配合）。
- **幻觉**：模型会生成流畅但不符合事实的内容；缓解手段包括 RAG、工具调用、后处理与人在回路。
- **与下游应用的关系**：可直接「零样本 / 少样本」使用，也可通过 Prompt、微调、RAG 等适配业务。

## Prompt 工程（提示工程）

**定义**：通过设计输入文本的结构、角色、约束、示例与输出格式，引导 LLM 稳定产出期望结果，而无需或尽量少改模型权重。

**常见技法**：
- **角色与任务说明**：明确身份、目标、受众与成功标准。
- **Few-shot**：在提示中给 1～若干正确输入输出示例，诱导模仿格式与推理方式。
- **Chain-of-Thought（CoT）**：要求「逐步推理」，提升算术、逻辑类任务表现（也会更长、更耗 token）。
- **结构化输出**：约定 JSON、字段名、枚举值，便于程序解析（可与 JSON mode / 约束解码配合）。

**与微调的关系**：Prompt 偏「推理时配置」；微调偏「改变参数」；二者常组合：小改行为用 Prompt，深度对齐风格/领域用语可考虑微调或 RAG。

## 微调流程与 LoRA / QLoRA

**全量微调（Full Fine-Tuning）**：更新全部权重，效果好但对算力、显存、数据量要求高，且易灾难性遗忘。

**LoRA（Low-Rank Adaptation）**：
- **思想**：冻结原模型大部分权重，仅在部分线性层旁路插入低秩矩阵 A·B（秩 r 远小于原维度），只训练这部分小参数。
- **效果**：显存与可训练参数大幅下降，便于单卡或多卡做领域适配；推理时可合并权重进原层，几乎不增加延迟。
- **典型流程**：准备指令/对话数据 → 选目标模块（常 attention 的 q/v 等）→ 设 rank、学习率、epoch → 训练 → 合并或保留适配器按需加载。

**QLoRA（Quantized LoRA）**：
- **思想**：基座模型以 4-bit（等）量化加载，降低显存占用；仍用 LoRA 训练适配器，在数值上通过反量化与梯度策略保持可训练性。
- **适用**：显存紧张、希望在消费级 GPU 上微调较大模型。
- **注意**：需关注量化与训练库的兼容性、学习率与秩的选择；极端情况下精度相对全精度+LoRA 可能有差距。

**小结**：微调解决「领域话术、格式、行为」等；通用知识与事实更新更常配合 RAG 或持续预训练，而非单靠小数据微调。

## RAG（检索增强生成，Retrieval-Augmented Generation）

**定义**：在生成前从外部知识库（向量库、文档库、图数据库等）检索与当前问题相关的片段，把检索结果作为上下文一并输入 LLM，使回答**可溯源、可更新**，减轻纯参数记忆带来的幻觉与过时。

**典型流水线**：
1. **索引**：文档切分（chunk）→ 嵌入模型向量化 → 存入向量索引（可辅以元数据过滤）。
2. **检索**：用户问题向量化 → 相似度检索（Top-K）→ 可选重排序（rerank）。
3. **生成**：将「检索到的片段 + 问题 + 系统指令」拼进 Prompt，由 LLM 生成答案；可要求引用片段编号或出处。

**与微调的边界**：RAG 负责「动态、可审计的事实与文档」；微调负责「语气、任务形态、工具使用习惯」等。生产系统常 RAG + 好 Prompt + 可选轻量 LoRA 一起用。

## Agent 基础范式（ReAct、Plan-and-Execute）

**定义**：在 LLM 之上用**循环控制流**把「推理」与「对外部世界的动作」（调用 API、查库、跑代码、点 UI 等）串起来，使模型不只在上下文里「说完即止」，而是**多步达成目标**。RAG 常作为 Agent 的一种**工具**（检索），Agent 则强调**决策与工具编排**。

### ReAct（推理与行动）

**是什么**：*ReAct* 这个名字来自英文 **Reason**（推理）和 **Act**（行动）。这类 **AI Agent** 指：模型在完成任务时**交替**做两件事——先用自然语言**推理**当前局面、下一步该做什么，再发出**可执行动作**（如调用搜索、计算器、数据库等工具）；运行时把工具的返回当作**观察**写回对话，模型再基于新信息继续推理与行动，直到给出最终答案。也就是中文常说的**「想一步、做一步、看结果、再想」**，而不是单轮里一次性「说完」。

**思路**：让模型交替输出**推理片段（Thought）**与**可执行动作（Action）**，环境或运行时返回**观察（Observation）**，再进入下一轮，直到给出**最终答案（Final Answer）**。

**典型循环**：Thought → Action（如调用某工具及参数）→ Observation（工具返回）→ Thought → …

**优点**：单轮 Prompt 难以覆盖的长链条任务可逐步纠偏；动作与观察把**事实**从模型参数里「拉」到当前上下文，减轻幻觉。

**代价与注意**：步数多则延迟与 token 成本高；需约束动作空间（允许的工具、参数模式）、超时与重试，并对危险操作做权限与人审。

### Plan-and-Execute（先规划再执行）

**思路**：分两阶段（或多阶段）：**规划器**先产出结构化计划（子任务列表、依赖、验收标准）；**执行器**按计划逐步调用工具或子 Agent 完成子任务；必要时根据失败或新信息**重规划（replan）**。

**与 ReAct 的对比**：ReAct 更「边想边做」，灵活但易在长任务上漂移；Plan-and-Execute 把「要做什么」先写清，利于**可审计、可并行子任务、人类改计划**；缺点是规划可能脱离实际，需执行反馈闭环。

**实践要点**：计划宜短步、可验证；执行层仍可内嵌小型 ReAct 处理单步不确定性；与 RAG、MCP、函数调用等组合时，明确**谁负责选工具、谁负责解析 Observation**。

**小结**：ReAct 适合工具多、环境反馈快的交互式任务；Plan-and-Execute 适合步骤清晰或可分解的项目型任务。生产上常见**混合**：粗粒度计划 + 细粒度 ReAct 执行 + RAG/工具兜底。
