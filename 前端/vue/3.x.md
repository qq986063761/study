# 和 vue2 的区别

## 核心架构变化
- **响应式系统重构**：通过 Proxy 重新设计响应式，渲染性能更好，支持动态属性监听
- **组合式 API**：让逻辑复用更好，代码组织也方便，解决了 vue2 中 mixin 的变量冲突和来源不清楚的问题
- **TypeScript 原生支持**：完全使用 TypeScript 重写，提供更好的类型推断和开发体验

## Composition API 比 Options API 好在哪里？
- 逻辑复用更方便：抽离逻辑，避免 mixin 命名冲突和来源不清晰问题。
- 代码组织更灵活：相同功能逻辑可以写在同一个 setup 区域，而不是分散在 data、methods、computed 等选项中，特别是复杂组件更清晰。
- TypeScript 支持更好：函数式写法天然适合类型推导。
- 但也需要注意：Composition API 不是完全替代，Options API 在简单场景下仍有可读性优势。

## <script setup> 相比普通 setup() 有什么优点？
- 更简洁：顶层变量和导入的组件自动暴露给模板，不需要 return。
- 编译时优化：模板编译时可以直接分析顶层绑定，更高效。
- 更好的 TypeScript 支持。
- defineProps/defineEmits 等编译器宏无需导入，提供更纯粹的类型声明方式。

## 性能优化
- **打包体积减少**：Tree-shaking 打包移除未使用代码，仅打包实际使用的模块
- **编译优化**：
  - 静态元素提升（Static Hoisting）
  - 虚拟节点静态标记（Patch Flag）
  - 事件处理函数缓存（Cache Handler）
- **SSR 渲染性能提升**：服务器端渲染速度提高 2-3 倍
- **Fragment 支持**：模版支持多个根节点，减少不必要的包装元素

## 模板和指令变化
- **v-if 和 v-for 优先级**：Vue 3 中 v-if 优先级高于 v-for
- **移除过滤器（Filters）**：不再支持 `{{ value | filter }}` 语法，推荐使用计算属性或方法
- **移除 `.sync` 修饰符**：使用 `v-model` 替代

## Vue3 生命周期相较于 Vue2 的变化？
- Vue2 的 beforeCreate 和 created 被 setup() 取代
- 其他生命周期改为 on 前缀：onBeforeMount、onMounted、onBeforeUpdate、onUpdated、onBeforeUnmount、onUnmounted 等，在 setup 中注册。
- 新增调试相关的 onRenderTracked、onRenderTriggered。
- 组合式 API 中没有 activated/deactivated 对应的 onActivated/onDeactivated 也保留。

## 新增特性
- **新内置组件**：
  - `Teleport`：可将组件内容渲染到 DOM 任意位置
  - `Suspense`：处理异步组件加载的占位内容
- **新指令**：`v-memo` 用于缓存模板片段
- **新 API**：`defineAsyncComponent`、`defineEmits`、`defineProps` 等

## 移除的特性
- **事件总线**：移除 `$on`、`$off`、`$once` 实例方法
- **过滤器**：完全移除过滤器功能
- **全局 API 变化**：`Vue.extend`、`Vue.component` 等全局 API 改为应用实例方法
- **`.sync` 修饰符**：使用 `v-model` 替代
- **`$children` 和 `$listeners`**：移除这些实例属性

# 性能提升详解

## Vue3 响应式原理

### 核心实现

Vue3 使用 **Proxy** 实现响应式，对整个对象进行代理，拦截对象的读取、修改、删除等操作。

当数据被读取时，会进行**依赖收集（track）**；当数据发生变化时，会**触发依赖更新（trigger）**，通知使用该数据的组件重新渲染。

---

### 为什么不用 Object.defineProperty？

Vue2 使用 `Object.defineProperty` 只能劫持已有属性，需要递归遍历对象初始化响应式，存在以下问题：

- 无法监听对象新增、删除属性（需要 `Vue.set`、`Vue.delete`）
- 无法监听数组索引和 `length` 的变化
- 初始化需要递归遍历整个对象，大对象性能较差

---

### Proxy 解决了什么问题？

- 可以监听对象新增、删除属性
- 可以监听数组索引和 `length` 的变化
- 代理整个对象，不需要为每个属性单独劫持
- 采用**懒代理**，只有访问嵌套对象时才创建代理，初始化性能更好
- 支持更多拦截操作（如 `get`、`set`、`deleteProperty` 等），扩展能力更强

---

### 面试总结（一分钟回答）

Vue3 使用 **Proxy** 重写了响应式系统，通过 **track（依赖收集）** 和 **trigger（派发更新）** 实现数据驱动视图。相比 Vue2 的 `Object.defineProperty`，Proxy 能监听对象新增、删除属性以及数组索引、`length` 的变化，同时采用懒代理减少初始化开销，因此功能更完善、性能更好。

## 编译时优化
- **Vue 2.x**：标记静态节点，优化 diff 过程
- **Vue 3.x** 编译优化：
  - **静态提升（Static Hoisting）**：所有静态节点提升到 render 方法外，只会在应用启动时创建一次
  - “预字符串化”：连续静态节点直接编译为字符串 innerHTML，减少虚拟节点比对。
  - **Patch Flag**：动态节点会被标记出变化的类型（如 class、text），只对比必要部分，跳过静态内容
  - **事件处理函数缓存**：避免每次触发都重新生成新函数
  - **Fragments 支持**：template 中可不要唯一根节点，减少包装元素
  - 最长递增子序列：更新列表时，尽量让更多的节点不动，只移动必须要动的，省去多余 DOM 操作。
  - Block Tree 靶向更新：模板里只盯着会变的地方更新，静止的部分完全不理，避免全树检查。

## diff 算法中 key 的作用是什么？
- `key` 用来标识虚拟节点的身份，帮助 Vue 在列表更新时识别哪些节点保持不变、哪些需要复用。
- **没有 key 的问题**：Vue 会按索引位置进行 diff，可能导致组件状态混乱、输入框值错乱等问题。
```vue
<!-- ❌ 没有 key，更新可能出错 -->
<li v-for="item in list">{{ item.name }}<input /></li>

<!-- ✅ 使用唯一的 key -->
<li v-for="item in list" :key="item.id">{{ item.name }}<input /></li>
```
- **最佳实践**：使用唯一、稳定的标识符（如 id），避免使用 index，特别是列表可能增删的场景。
- Vue 3 中的最长递增子序列算法：当存在 key 时，Vue 会计算最长递增子序列，最小化 DOM 操作次数。

## 代码体积优化
- **Tree-shaking**：基于 ES Module 的静态结构，函数只有用到才会打包，编译阶段静态分析，未引入的模块会被移除;vue3的组合式代码组织能更好的支持打包工具的 Tree-shaking
- **模块化设计**：移出一些不常用的 API，按需引入
- **体积减少**：相比 Vue 2 减少约 41% 的打包体积

# 新组件详解

## Fragment（片段）
- **作用**：可以不用根标签，内容会包含在 Fragment 虚拟元素中
- **优势**：减少标签层级和内存占用
- **使用场景**：多根节点组件，避免不必要的包装元素

## Teleport（传送门）
- **作用**：可将组件 HTML 内容渲染到 DOM 中的任意位置
- **使用场景**：模态框、通知、下拉菜单等需要脱离当前组件层级的场景
- **语法**：`<Teleport to="body">...</Teleport>`

## Suspense（异步组件）
- **作用**：处理异步组件加载时的占位内容，优化用户体验
- **使用场景**：懒加载组件、异步数据获取等
- **语法**：配合 `defineAsyncComponent` 使用

## 异步组件的用法和好处？
- 用法 defineAsyncComponent
```ts
const AsyncComp = defineAsyncComponent(() => import('./Comp.vue'))
const AsyncComp = defineAsyncComponent({
  loader: () => import('./Comp.vue'),
  loadingComponent: Loading,
  errorComponent: Error,
  delay: 200,
  timeout: 3000
})
```
- 按需加载，减少首屏体积

## keep-alive 在 Vue 3 中的用法和区别
- **作用**：缓存组件状态，避免重复创建销毁，提高性能。
- **Vue 3 新写法**：配合路由使用 `v-slot="{ Component }"` 解构形式
```html
<router-view v-slot="{ Component }">
  <keep-alive>
    <component :is="Component" />
  </keep-alive>
</router-view>
```
- **include / exclude**：指定哪些组件需要缓存（按组件 name）
```html
<keep-alive include="CompA,CompB" exclude="CompC" :max="10">
  <component :is="dynamicComp" />
</keep-alive>
```
- **max**：缓存组件数量上限，超出后删除最久未使用的组件（LRU 策略）。
- **缓存注意**：被缓存的组件生命周期不再触发 onBeforeUnmount/onUnmounted，而是触发 onDeactivated（失活）；再次激活时触发 onActivated。
- **Vue 3 区别**：Vue 2 中使用 `<keep-alive>` 只需在 `<router-view>` 外层包裹，Vue 3 推荐使用 `v-slot` 获得更好的控制粒度。

# 生命周期详解

## 选项式 API 生命周期
- **beforeCreate**：初始化选项之前触发
- **created**：初始化完成后触发，数据可访问，组件未挂载到 DOM
- **beforeMount**：创建和插入 DOM 节点之前触发
- **mounted**：DOM 节点渲染完成后触发
- **beforeUpdate**：虚拟 DOM 重新渲染和打补丁之前触发
- **updated**：虚拟 DOM 重新渲染和打补丁之后触发
- **beforeUnmount**：组件被销毁前触发（Vue 2 中为 beforeDestroy）
- **unmounted**：组件销毁后触发（Vue 2 中为 destroyed）

## 组合式 API 生命周期
- **onBeforeMount**：需要显式导入
- **onMounted**：需要显式导入
- **onBeforeUpdate**：需要显式导入
- **onUpdated**：需要显式导入
- **onBeforeUnmount**：需要显式导入
- **onUnmounted**：需要显式导入
- **onActivated**：keep-alive 组件激活时
- **onDeactivated**：keep-alive 组件失活时

# 组合式 API 核心概念

## ref 和 reactive 的区别

### 解决什么问题？

Vue3 需要让数据具有响应式能力，根据数据类型不同，提供了 `ref` 和 `reactive` 两种方式。

### ref

* 主要用于**基本类型**，也可以包裹对象
* 使用时需要 `.value`（模板中自动解包）
* 可以直接替换整个值
* 包裹对象时，对象内部依然是响应式的

适合：

* number、string、boolean
* 单个状态
* 需要整体替换的数据

### reactive

* 只能代理对象（Object、Array、Map、Set 等）
* 不需要 `.value`
* 不建议直接替换整个对象，否则会丢失响应式
* 解构会丢失响应式，可使用 `toRefs` 或 `toRef`

适合：

* 表单
* 用户信息
* 列表、复杂对象

### 面试总结

`ref` 更适合基本类型或独立状态，`reactive` 更适合复杂对象；对象解构需要配合 `toRefs` 保持响应式。

---

# computed、watch、watchEffect 的区别

### 解决什么问题？

当数据变化时，有三种不同需求：

* **需要得到一个计算后的新值** → computed
* **需要监听变化执行副作用（请求接口、DOM 操作等）** → watch
* **依赖很多，不想手动指定监听源** → watchEffect

---

### computed

* 根据已有数据计算新数据
* 有缓存，依赖不变不会重复计算
* 必须有返回值
* 不适合处理副作用

适合：

* 价格计算
* 过滤列表
* 格式化数据显示

---

### watch

* 手动指定监听的数据
* 可以获取新值、旧值
* 支持 `immediate`、`deep` 等配置
* 可以处理异步请求、副作用

适合：

* 请求接口
* 数据校验
* 监听路由变化

---

### watchEffect

* 自动收集函数内部使用到的响应式数据
* 默认立即执行一次
* 获取不到旧值
* 不需要手动指定监听源

适合：

* 简单副作用
* 依赖较多的场景

---

### 对比

| 特性     | computed | watch | watchEffect |
| ------ | -------- | ----- | ----------- |
| 是否缓存   | ✅        | ❌     | ❌           |
| 获取新旧值  | ❌        | ✅     | ❌           |
| 默认立即执行 | ❌        | 可选    | ✅           |
| 自动收集依赖 | ✅        | ❌     | ✅           |
| 处理副作用  | ❌        | ✅     | ✅           |

### 面试总结

* **computed**：计算新数据
* **watch**：监听指定数据变化
* **watchEffect**：自动监听依赖变化

---

# 自定义 Composable

### 解决什么问题？

将多个组件中重复的业务逻辑抽离出来，实现**逻辑复用**，而不是复用模板。

例如：

* 请求接口
* 倒计时
* 鼠标监听
* 分页
* 本地存储

都可以封装成 `useXXX`。

---

### 设计规范

* 命名统一以 `use` 开头，如 `useFetch`、`useMouse`
* 返回响应式数据和方法
* 内部可使用 `ref`、`reactive`、生命周期等 API
* 一般在 `setup` 同步阶段调用

### 面试总结

Composable 本质就是把可复用的状态和逻辑封装成函数，提高代码复用性。

---

# props / emit（属性与事件）与单向数据流

### 解决什么问题？

实现父子组件通信。

* 父组件通过 **props** 向子组件传递数据
* 子组件通过 **emit** 通知父组件修改数据

---

### 什么是单向数据流？

数据只能：

```
父组件
   │
 props
   │
子组件
```

子组件不能直接修改 `props`。

如果修改：

* 会导致数据来源混乱
* 父组件重新渲染时会覆盖修改

因此需要：

```
emit('update')
```

通知父组件修改数据。

### 面试总结

Props 负责传数据，Emit 负责通知修改，保证数据只有一个来源，方便维护。

---

# provide / inject（提供与注入）

### 解决什么问题？

解决跨层级组件通信，避免 **Prop Drilling（props 层层传递）**。

---

### 优点

* 不需要中间组件传递 props
* 降低组件耦合

### 缺点

* 数据来源不够直观
* 调试比 props 困难
* 不适合普通父子通信

---

### 如何避免 key 冲突？

推荐使用 `Symbol` 作为 key。

```ts
const ThemeKey = Symbol()

provide(ThemeKey, theme)

const theme = inject(ThemeKey)
```

### 面试总结

provide/inject 适合祖先组件向后代共享数据，不适合替代 props。

---

# v-model

### 解决什么问题？

实现父子组件双向绑定。

父组件修改数据，子组件同步更新；

子组件修改数据，通过 `emit` 通知父组件更新。

---

### Vue3.0~3.3

底层：

* `modelValue`
* `update:modelValue`

---

### Vue3.4+

使用：

```ts
const model = defineModel()
```

无需手动声明：

* props
* emit

代码更简洁。

---

### 多个 v-model

支持：

```vue
<Child
  v-model:title="title"
  v-model:count="count"
/>
```

### 面试总结

Vue3 的 `v-model` 本质还是 **props + emit**，3.4 以后通过 `defineModel` 简化了写法。

---

# 插槽（Slot）

### 解决什么问题？

让父组件决定子组件部分内容如何渲染，提高组件复用性。

---

### 默认插槽

父组件传入普通内容。

```vue
<slot />
```

---

### 具名插槽

一个组件有多个插槽区域。

```vue
<slot name="header" />
```

---

### 作用域插槽

子组件向父组件暴露数据。

例如：

```vue
<slot :item="item" />
```

父组件可以拿到：

```vue
<template #default="{ item }">
```

适合：

* Table
* List
* Form

等需要自定义渲染的组件。

### 面试总结

* 默认插槽：传内容
* 具名插槽：多个区域
* 作用域插槽：子组件向父组件传数据，实现模板自定义

# 状态管理

## Pinia 比 Vuex 好在哪里
- 模块化设计更简洁：不再需要 mutation，直接通过 state 和 actions 管理数据。
- 完美的 TypeScript 支持，类型推断无须额外工具。
- 没有嵌套模块、命名空间等概念，每个 store 独立声明，扁平化。
- 支持 Composition API 风格的 setup store 和 Options API 风格的 defineStore 两种写法。
- 轻量（约 1KB），可直接用于 Vue2/Vue3，去除了 context 等冗余概念。
- 安装为 Vue Devtools 提供更好的支持。

## Vuex 在 Vue3 中仍可用吗？怎么看官方推荐 Pinia
- Vuex 4 支持 Vue3，但官方已推荐 Pinia 作为新的状态管理方案。
- Vuex 依然可用，适合旧项目迁移，但新项目优先选择 Pinia。
- 面试中可能需要对比两者的设计理念差异：Vuex 基于 Flux 结构（state/mutation/action/module），Pinia 更接近组件化的组合思想。

## 重要 API 变化
### 新增 API
- **defineProps**：定义组件 props
- **defineEmits**：定义组件 emits
- **defineExpose**：暴露组件实例方法
- **defineAsyncComponent**：定义异步组件
- **withDefaults**：为 defineProps 提供默认值
- **v-memo**：缓存模板片段

### 全局 API 变化
- **Vue.createApp()**：替代 `new Vue()`
- **app.component()**：替代 `Vue.component()`
- **app.directive()**：替代 `Vue.directive()`
- **app.use()**：替代 `Vue.use()`

## 迁移注意事项
### 破坏性变化
- **过滤器移除**：使用计算属性或方法替代
- **事件总线移除**：使用 provide/inject 或状态管理
- **$children 移除**：使用 ref 获取子组件
- **$listeners 移除**：使用 v-on="$attrs" 替代
- **.sync 修饰符移除**：使用 v-model 替代

### 推荐迁移路径
1. **渐进式迁移**：可以同时使用选项式 API 和组合式 API
2. **使用 `<script setup>`**：简化组合式 API 语法
3. **使用 Pinia**：替代 Vuex 进行状态管理
4. **使用 Vite**：替代 Vue CLI 获得更好的开发体验
